# 💾 TrueNAS iSCSI 스토리지 구성

## 1. 개요 및 프로젝트 내 역할

주센터의 가상화 환경은 TrueNAS를 iSCSI 기반 공유 스토리지로 사용합니다. ESXi 호스트는 TrueNAS가 제공하는 iSCSI LUN을 데이터스토어로 마운트하여, 운영 DB·OS 디스크와 백업·복제본을 저장합니다.

블록 스토리지(iSCSI)를 선택한 이유는, ESXi 데이터스토어로 직접 마운트하여 VM 디스크(VMDK)를 올릴 수 있고, VMFS 기반의 스냅샷·vMotion 등 가상화 기능과 자연스럽게 연동되기 때문입니다.

<br />

## 2. 스토리지 티어링 설계

저장 대상의 성격에 따라 **NVMe와 HDD로 티어를 분리**했습니다.

| 티어 | 용도 |
| --- | --- |
| NVMe | 운영 DB, OS 디스크 — 낮은 지연·높은 IOPS가 필요한 영역 |
| HDD | 백업, 복제본 — 용량 위주이고 지연에 둔감한 영역 |

운영 DB와 OS는 응답 지연이 거래 처리에 직접 영향을 주므로 NVMe에 배치하고, 백업·복제본은 순차 쓰기 위주에 용량이 중요하므로 HDD에 배치하여 비용과 성능의 균형을 맞췄습니다.

<br />

## 3. 주요 설정

### 3-1. iSCSI 구성 흐름

1. TrueNAS에서 zvol(블록 볼륨)을 생성하고 iSCSI Target / Extent / Portal을 구성합니다.
2. 티어 분리를 위해 NVMe 풀과 HDD 풀을 각각 별도 zvol로 생성합니다.
3. ESXi에서 소프트웨어 iSCSI 어댑터를 활성화하고 TrueNAS Portal을 Target으로 등록합니다.
4. 디스커버리된 LUN을 VMFS 데이터스토어로 포맷하여 마운트합니다.

### 3-2. 단일 경로 환경의 제약 인지

이 환경의 iSCSI는 **단일 경로(single path)** 로 구성되어 있습니다. 멀티패스(MPIO) 이중화가 없으므로, 스토리지 네트워크에 변경이 생기면 경로가 끊겼다 붙는 **flapping** 이 곧바로 I/O 중단으로 이어집니다.

<br />

## 4. 운영 시 주의 — 스토리지 네트워크 변경 절차

스토리지 네트워크(iSCSI) 구성을 변경하기 전에는 **반드시 해당 LUN을 사용하는 DB를 graceful하게 종료**해야 합니다.

> ⚠️ **TRAP**
> 단일 경로 환경에서 iSCSI 경로 flapping이 발생하면, ESXi가 디스크 I/O 타임아웃을 감지하고 해당 데이터스토어를 일시 차단합니다. 이때 DB가 올라가 있던 게스트 VM의 파일시스템(XFS)이 **I/O 타임아웃 → 파일시스템 셧다운** 으로 직결되어, DB가 비정상 종료되거나 데이터 정합성이 깨질 수 있습니다.
>
> 따라서 iSCSI 네트워크 변경(포트그룹 이전, VLAN 변경, 스위치 작업 등) 전에는 해당 DB를 먼저 정상 종료한 뒤 작업하고, 작업 완료 후 경로 안정화를 확인하고 나서 DB를 재기동합니다.

<br />

## 5. 동작 흐름 및 트러블슈팅

### 정상 동작 흐름

1. ESXi가 TrueNAS iSCSI Target에 연결하여 LUN을 데이터스토어로 마운트합니다.
2. 운영 DB·OS VM은 NVMe 풀 기반 데이터스토어에, 백업·복제본은 HDD 풀 기반 데이터스토어에 배치됩니다.
3. VM은 iSCSI 경로를 통해 블록 I/O를 수행합니다.

### 트러블슈팅

| 증상 | 원인 | 조치 |
| --- | --- | --- |
| DB VM의 XFS 파일시스템이 read-only/셧다운 상태로 전환 | 단일 경로 iSCSI flapping으로 인한 I/O 타임아웃 | 경로 안정화 확인 → 해당 VM 재기동 → 파일시스템·DB 정합성 점검 |
| 스토리지 네트워크 작업 중 데이터스토어 일시 차단 | 작업 전 DB 미종료 | 작업 전 DB graceful 종료 절차 준수 (사전 예방) |
| 운영 DB 응답 지연 | 티어 배치 오류(DB가 HDD 풀에 위치) | 운영 DB·OS는 NVMe 풀로 재배치 |
